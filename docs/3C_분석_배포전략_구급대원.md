> 작성일: 2026.04.24 · 작성자: 이진영(미스터슈어, 팀 AML)
> 형식: 3C(Customer / Company / Competitor) + USP 한 줄 정의
> 목적: **B2G/B2B 수직 확장(Vertical)** — "119 구급대원 · 민간 이송 구급대원"을 타겟으로 한 파일럿 배포 전략을 1페이지로 확정한다.

## 0. 요약 — "한 줄 앱 정의" (USP)

> **"이송 중 병원을 정하기 위해 전화 4~5통을 돌리던 구급대원에게, 동시 자동 전화 + 실시간 수용 가능성 + 출동 자동 로그·PDF 보고서를 한 화면으로 주는, 설치 없는 PWA."**

- **핵심 타겟**: 경력 3~10년차 **119 구급대원** (구급지도반·전담 구급반)
- **핵심 문제**: 이송 중 "병원 선정 + 수용 확정 + 일지 기록"을 **한 손으로 동시에** 해야 하는 극단적 멀티태스킹
- **핵심 차별**: 공공 시스템이 제공하지 못하는 **"실시간 수용 가능성 + 동시 자동 전화 + 업무 로그 자동 기록"** 의 **단일 화면 흐름**

## 1. 🟦 Customer — 가장 중요한 타겟 고객 1명

### 페르소나 "김태호, 35세 · 소방교 · 수도권 광역시 119안전센터 전담 구급반 · 경력 8년"

#### 1-1. 어떤 사람인가? (나이 · 직업 · 상황)

| 항목 | 내용 |
|---|---|
| **나이/성별** | 35세 남성 (30~45세 구급대원 중위값) |
| **직책/계급** | 소방교(중견급) · 전담 구급반 팀장 또는 1급 응급구조사 |
| **소속** | 수도권 광역시 소방서 119안전센터, 3교대 근무 (주간·야간·비번) |
| **월 출동 건수** | 약 180~220건 / 1인 (센터당 하루 8~15건) |
| **자격** | 1급 응급구조사 또는 간호사 면허 + 전문응급처치 자격 |
| **디지털 숙련도** | 중상: MDT(차량 태블릿) 익숙, 스마트폰 기반 업무 앱 거부감 낮음 |
| **신체·정신 상태** | 교대근무로 만성 피로, 심리 소진 고위험군 (PTSD 리스크) |
| **주 사용 디바이스** | 차량 MDT(거치형 태블릿, Android) + 개인 업무폰(iOS/Android) |
| **네트워크 환경** | LTE/5G + 터널·지하주차장 음영 빈번 (오프라인 폴백 필수) |
| **업무 환경 특이성** | 한 손은 환자 고정·혈압계, 다른 손은 장비·무전기 → 터치 정확도 낮음, 음성 UX 선호 |

#### 1-2. 가장 큰 Pain Point 한 문장은?

> **"이송 중 흔들리는 차 안에서, 한 손으로 환자를 잡고 다른 손으로 병원 4~5곳에 전화를 돌리며, 동시에 무전으로 CAD에 보고하고 머릿속에 구급일지를 작성하고 있다."**

세부 고통 (현장 인터뷰 기반 요약)
- **"응급실 뺑뻉이는 환자보다 우리가 먼저 겪는다"** — 1시간째 전화만 돌리고 있는 경우 존재
- **구급활동일지 수기 작성 부담** — 귀소 후 2~3건씩 밀려 야간근무 추가로 이어짐
- **활력징후·투약 시간 기록 누락** — 손으로 메모하다 분실/오기입
- **이송 병원 확정 후 교통 상황 재확인**을 위해 앱 여러 개를 전환해야 함
- **MDT 시스템은 관할·기본 정보만 제공**, 실시간 수용 가능성 미제공
- **환자 인계 후 병원측 KTAS와 현장 KTAS 불일치** 시 소명 자료 부재

#### 1-3. 언제 이 앱이 필요한가? (JTBD · Job to Be Done)

> **"[When] 중증 이상 환자를 이송 중 병원 선정·수용 확정을 3분 안에 끝내야 할 때**
> **[I want to] 한 손 · 음성으로 동시에 여러 병원에 수용 가능성을 자동 확인하고, 출동~인계 전 과정이 자동으로 기록되게 하고 싶다**
> **[So I can] 환자 곁에 손과 눈을 두고, 골든타임 내에 적정 병원으로 이송하며 귀소 후 일지 작성 부담을 덜 수 있다."**

**대표 트리거 상황(4가지)**
1. **급성 흉통·심정지 의심 환자** — 심혈관 센터(24시간 PCI) 중 수용 가능한 곳 즉시 확인 필요
2. **뇌졸중 의심(FAST 양성)** — 혈전용해술(IV-tPA) 가능 병원 중 4.5시간 윈도우 확인
3. **중증외상(사고·낙상)** — 권역외상센터 수용 확정 필요, 평일 야간 뺑뻉이 최다
4. **소아 경련·호흡곤란** — 소아응급전문 병원은 더 제한적, 야간에 가용 병원 희소

**서브 JTBD(귀소 후)**
- "출동 기록을 기억만으로 재구성하지 않고, 자동 수집된 타임라인으로 일지를 5분 내 완성하고 싶다"

## 2. 🟩 Company — 우리 앱의 핵심 강점 · 약점

#### 2-1. 경쟁 대비 가장 강한 기능 1가지는?

> **"출동·현장·이송·인계·귀소 5단계의 타임스탬프를 자동 기록하고, 동시에 3~5곳 자동 전화로 수용 가능성을 확정한 뒤, PDF/Excel 보고서로 원터치 내보내는 '단일 화면 워크플로우'.**"

근거
- **MDT 차량 태블릿**: 관할·지도·무전 중심, 수용 확정·일지 자동화 약함
- **E-Gen 구급대원 기능**: 정보 열람 수준, 상호작용·기록 자동화 부재
- **e-구급활동일지 시스템**: 귀소 후 **수동 입력** 전제 — 현장 실시간 기록 불가
- **우리**: **실시간 수용 확정 + 현장 데이터 자동 수집 + 보고서 자동 생성** 의 3-in-1

강점 Top 5 (중요도 순)
1. **병원 선정의 의사결정 시간 단축** — 순차 통화(3~5분) → 동시 자동 전화(30~60초)
2. **구급활동일지 자동화** — 출동~인계 타임스탬프 + 환자 상태·수용 병원 자동 첨부
3. **PDF/Excel 한 번에 내보내기** — 기관 결재·통계 시스템 연계 용이
4. **PWA 즉시 실행** — 기존 MDT 외 개인 단말에서도 동일 경험 (원격·차량 외 대응 가능)
5. **오프라인 폴백** — 터널·지하 음영에서도 최근 데이터로 최소 판단 지원

#### 2-2. 파일럿 기간 내 반드시 개선해야 할 약점?

| 약점 | 영향 | 파일럿 내 대응 |
|---|---|---|
| **W1. 소방청 공식 시스템(NEMC·CAD)과의 연계 미비** | 이중 입력 우려 | 파일럿에서는 **보조 도구**로 포지셔닝 + 일지 CSV 포맷을 기관 양식에 맞춤 |
| **W2. 자동 전화의 법적·조직적 리스크** | 기관 승인 장벽 | 수신 병원 안내멘트 공식화, 지역 응급의료센터장 사전 협의, 로그 전체 공개 |
| **W3. 공공데이터 API 지연·정확도** | 현장 신뢰 훼손 | 캐시 + "N분 전 데이터" 라벨 + "전화 우선" 권고 플래그 |
| **W4. 차량 내 한 손 UX** | 터치 오조작 | 큰 히트 영역(56pt+), 음성 명령("3번 전화"), 햅틱·보이스 피드백 |
| **W5. 교대근무 특성상 학습시간 부족** | 사용 정착 지연 | 3분 영상 온보딩 + 현장 코치(동료 전파) 모델 + 1페이지 카드형 매뉴얼 |
| **W6. 개인정보·의료법 리스크** | 도입 심의 통과 실패 | 환자 식별정보 서버 미저장, 익명 해시 로그, 법률·의료법 자문 1회 |
| **W7. 1인 팀 지원 한계** | SLA 보장 어려움 | 파일럿 규모 제한(1개 센터 20명 이하) + Slack 핫라인 + 월 1회 현장 간담회 |

#### 2-3. 핵심 기능 1개에 집중한다면?

> **"이송 중 '병원 확정' 까지의 의사결정 시간을 30초 안에 끝내주는 동시 자동 전화 + 실시간 수용 가능성 확인."**

이유
- 페르소나 Pain Point(1-2)의 **본질적 병목**
- 이 한 기능이 **"분 단위 시간 절약" → "골든타임 단축" → "의료 결과 개선"** 이라는 파일럿 성과 지표로 직접 연결
- 구급활동일지 자동화는 **이 기능의 부산물(자동 수집된 로그)** 로 자연 확장
- 기관이 가장 납득하기 쉬운 ROI — "1건당 평균 2분 단축 × 월 200건 = 월 6.7시간/인" 의 정량화 가능

## 3. 🟪 Competitor — 사용자가 지금 쓰는 대안

#### 3-1. 직접 경쟁 앱/시스템 Top 2

| # | 서비스 | 제공 주체 | 포지션 | 사용률(추정) |
|---|---|---|---|---|
| **① MDT + 관할 CAD(119종합상황실 연동 시스템)** | 소방청·시도 소방본부 | 차량 태블릿 기반 출동 정보·지도·환자 기본정보 | 전 구급차 필수 탑재 (100%) |
| **② e-구급활동일지(전자구급일지 시스템)** | 소방청 국가정보시스템 | 귀소 후 PC 기반 일지 작성·제출 | 100% (의무) |

#### 3-2. 사용자가 지금 쓰는 **간접 대안** (진짜 경쟁자)

| # | 대안 | 사용자 행동 |
|---|---|---|
| **① 응급실 전화번호부 + 수기 전화 돌리기** | 가장 흔한 디폴트 — 가까운 순 3~5곳을 순차 호출 |
| **② 무전으로 종합상황실에 "수용 가능 병원 조회" 요청** | 상황실 부하 시 대기 시간 증가, 음성 품질 한계 |
| **③ 병원별 응급실 당직실 직통번호 개인 수첩·메모앱** | 정확도 낮고 개인 종속적 |
| **④ 동료 구급대원 그룹 카카오톡·텔레그램** | "지금 A병원 수용돼?" 물어보기 — 공식 아님 |
| **⑤ E-Gen 모바일 앱 조회** | 정보 확인만, 전화·기록 불가 → 결국 ①로 복귀 |

**전략적 시사점**: 진짜 적은 MDT/e-구급일지 같은 **공식 시스템**이 아니라 **"손으로 전화 돌리기 + 수기 메모 + 귀소 후 재구성"** 의 **파편화된 개인 업무 습관**. 공식 시스템과 공존하는 **"현장 보조 도구"** 포지셔닝이 핵심.

#### 3-3. 그 대안의 가장 큰 불만은?

> **"이송 중 전화를 돌리느라 환자에서 손을 떼야 하고, 기록은 귀소 후 기억에 의존해 재구성해야 한다."**

구체적 불만 5선 (구급대원 커뮤니티·논문·블로그 기반 요약)
1. **"순차 전화는 너무 느리다"** — 병원마다 1~2분 소요, 5곳이면 10분 허비
2. **"환자에서 손을 떼야 한다"** — 활력징후 변화 감지 지연
3. **"기록이 기억에 의존한다"** — 투약·활력·시간 누락 → 일지 품질 저하
4. **"병원 KTAS와 현장 KTAS 불일치 시 근거가 없다"** — 다툼·민원 발생
5. **"MDT와 개인폰·e-구급일지 시스템이 따로 논다"** — 이중·삼중 입력

**우리의 즉답**: 5개 중 ①②③⑤ 을 바로응급실이 **구조적으로 해결**. ④는 증거 로그·스크린샷 기능으로 대응.

## 4. 🟨 USP — 한 줄 앱 정의 완성

#### 4-1. 핵심 타겟

> **"경력 3~10년차 119 전담 구급대원"** (1차) + **"권역외상센터 · 민간 이송업체 구급대원"** (확장)

#### 4-2. 핵심 문제

> **"이송 중 병원 확정과 업무 기록을 동시에 한 손으로 해결해야 하는데, 현재는 순차 전화와 수기 메모로 2~5분을 소모하고 있다."**

#### 4-3. 핵심 기능 + 차별 포인트 (한 문장)

> **"실시간 수용 가능성 + 동시 자동 전화 + 출동~인계 타임스탬프 자동 기록 + PDF/Excel 보고서 자동 생성을, 설치가 필요 없는 PWA 한 화면에서 제공한다."**

#### 4-4. 1줄 포지셔닝 스테이트먼트

> **For** 이송 중 병원 선정·수용 확정·일지 기록을 동시에 해야 하는 119 구급대원,
> **BaroER (Paramedic Mode)** is 설치 없이 실행되는 구급대원 전용 보조 PWA,
> **that** 실시간 수용 가능성 조회·동시 자동 전화·출동 타임스탬프 자동 기록·PDF 보고서 생성을 **한 화면** 에서 제공한다.
> **Unlike** 순차 전화·수기 메모·귀소 후 재구성의 파편화된 현재 업무 흐름,
> **we** "분 단위 시간 절약 + 기록 품질 향상" 을 동시에 달성하는 **유일한 현장 보조 도구** 다.

## 5. 배포 전략 시사점 (3C → B2G Go-To-Market)

### 5-1. 채널 전략 (Where to Launch)

| 단계 | 채널 | 접근 방식 | KPI |
|---|---|---|---|
| **Phase 0. 현장 인터뷰 & 공동 디자인** (출시 전 2~4주) | 현직 구급대원 커뮤니티(카페·오픈채팅) · 응급구조학과 동문 네트워크 | 심층 인터뷰 10명 + 섀도잉 2회 | 페르소나·JTBD 검증, 피드백 50건 |

| **Phase 2. 시 소방본부 공식 파일럿** (8~16주) | 시도 소방본부 기획·정책 라인 · 국립소방연구원 | 파일럿 성과 리포트 제출(시간 단축·일지 품질) | 공식 MOU 1건, 1개 지역 3~5센터 확대 |

### 5-2. 메시지 프레임 (What to Say)

**1줄 메인 카피 (현장용)**
> **"전화 4~5통을 30초로. 일지 작성 30분을 5분으로."**

**보조 카피 (공식 제안서용)**
- "MDT를 대체하지 않습니다. MDT가 비어 둔 '수용 확정과 현장 기록'의 공백을 채웁니다."
- "파일럿 1개 센터, 월 200건 기준 1인당 약 **6.7시간**의 현장 시간을 돌려드립니다."
- "오픈소스로 공개되어 소방청 레거시와 공존하며, 개인정보는 서버에 저장하지 않습니다."

**수치 기반 피치**
> "병원 선정 통화 시간 평균 4분 → **45초** (−82%)
> 구급활동일지 작성 평균 15분 → **5분** (−66%)
> 현장 기록 누락률 **>50%** → 타임스탬프 자동화로 구조적 제거"

### 5-3. 회피해야 할 메시지 (Don't)

- ❌ **"MDT·e-구급일지 대체"** — 조직적 저항 유발. "보조·공백 보완"으로 포지셔닝
- ❌ **"AI 자동 진단"** — 응급구조사의 현장 판단 영역을 침범, 의료법 리스크
- ❌ **"병원 수용 보장"** — 공공데이터 기반이므로 100% 보장 불가
- ❌ **"개인 데이터 수집"** — 환자 식별정보는 서버 미저장 원칙을 최상단 노출

### 5-4. 파일럿 지표 (센터 단위 90일 목표)

| 지표 | 목표 |
|---|---|
| 활성 사용자 (주 1회 이상) | 센터 내 15명 이상 |
| 평균 병원 선정 통화 시간 | 4:00 → **0:45** |
| 일지 작성 소요 시간 | 15:00 → **5:00** |
| 일지 핵심 항목 누락률 | < 5% (자동 타임스탬프 기반) |
| 자동 전화 오결 민원 | 0건 |
| NPS(현장 만족도) | ≥ 40 |
| 기관 성과 보고서 | 1건 (Phase 2 진입 가산점) |

## 9. 두 페르소나 비교표 (참고)

| 관점 | 1차: 일반 보호자 (박지영) | 2차: 구급대원 (김태호) |
|---|---|---|
| 사용 빈도 | **희박**(평생 1~5회) | **빈번**(일 5~15건) |
| 중요 지표 | 바이럴 계수 K, 공유율 | 시간 단축, 일지 품질, NPS |
| 채널 | 유튜브 Shorts · 맘카페 · SEO | 소방서 · MOU · 기관 제안서 |
| 결정 권한 | 개인 | 조직(승인 필요) |
| 매출 모델 | 장기적 공공 라이선스·프리미엄 | 기관 조달·라이선스·교육 |
| 학습 곡선 허용 | 0 (응급) | 3분 (교대 브리핑 시간) |
| 브랜드 톤 | 따뜻한·안심 | 전문적·절제된 |
| 핵심 메시지 | "설치 없이, 링크 한 번" | "전화 4통을 30초로" |
| 주 우선 기능 | 실시간 응급실 + 자동 전화 | 동시 자동 전화 + 업무 로그 자동화 |

> **두 세그먼트는 상호 배타적이지 않다.** 오히려 1차(B2C) 바이럴로 브랜드 인지도를 확보하고, 2차(B2G) 파일럿으로 신뢰·실증을 축적하는 **쌍방향 플라이휠** 이 바로응급실 Go-To-Market의 핵심 가설이다.
