## 1. 프로젝트 배경 및 목적

### 1.1 배경
응급 상황 발생 시 환자를 수용할 수 있는 가까운 응급실을 찾는 과정에서 시간 지연이 발생하고 있다. 특히 응급실 만상(滿床) 상황이 빈번하여, 여러 병원에 전화로 확인해야 하는 비효율이 존재한다. 이러한 문제를 해결하기 위해 실시간 응급실 수용 정보를 기반으로 최적의 응급실을 신속하게 안내하는 앱이 필요하다.

### 1.2 목적
- 응급 환자의 상태에 맞는 수용 가능한 인근 응급실을 **실시간으로 탐색**
- **지도·리스트** 두 가지 출력 형태로 직관적 안내 제공
- **음성 입력** 지원으로 긴급 상황에서도 빠른 조작 가능
- **자동 전화 확인** 기능으로 수용 여부를 동시 다발적으로 확인
- **구급대원 전용 기능**으로 업무 효율화 지원

### 1.3 목표 사용자
| 사용자 유형 | 설명 | 주요 사용 시나리오 |
|-------------|------|-------------------|
| 일반 사용자 | 응급 상황에서 직접 응급실을 찾아가는 일반 시민 | 가족·지인의 응급 상황, 본인 부상/질병 |
| 구급대원 | 119 구급대원, 소방 구급 인력 | 환자 이송, 응급실 수용 확인, 업무 보고서 작성 |

## 2. 핵심 요구사항

### 2.1 사용자 인증 (FR-AUTH)
| 항목 | 내용 |
|------|------|
| FR-AUTH-01 | 최초 설치 시 회원가입/로그인 화면 표시 |
| FR-AUTH-02 | 이메일, 비밀번호, 닉네임 입력으로 회원가입 |
| FR-AUTH-03 | 로그인 성공 시 토큰 저장, 이후 앱 실행 시 자동 로그인 (로그인 화면 미표시) |
| FR-AUTH-04 | 사용자 유형 선택 (일반 사용자 / 구급대원) |
| FR-AUTH-05 | 구급대원은 소속기관 인증 추가 (기관코드 입력) |

### 2.2 환자 상태 입력 (FR-INPUT)
| 항목 | 내용 |
|------|------|
| FR-INPUT-01 | 환자 증상 키워드 선택 (두통, 흉통, 외상, 호흡곤란 등) |
| FR-INPUT-02 | 상세 증상 텍스트 입력 |
| FR-INPUT-03 | 음성 입력 지원 (STT 변환) |
| FR-INPUT-04 | 환자 기본 정보 입력 (성별, 연령대, 의식 상태) |
| FR-INPUT-05 | 중증도 자동 분류 (입력 정보 기반) |

### 2.3 응급실 검색 및 표시 (FR-SEARCH)
| 항목 | 내용 |
|------|------|
| FR-SEARCH-01 | 현재 위치 기반 반경 내 응급실 자동 검색 |
| FR-SEARCH-02 | 공공데이터 API 연동 실시간 응급실 수용 정보 조회 |
| FR-SEARCH-03 | 지도 화면 모드: 수용 여부 색상 마커 표시 (🟢수용가능 / 🟠혼잡 / 🔴불가) |
| FR-SEARCH-04 | 리스트 화면 모드: 병원명, 거리, 예상소요시간, 병상수, 대응 의사 유무, 응급실 전화번호, 길안내 버튼 |
| FR-SEARCH-05 | 지도/리스트 모드 전환 토글 |
| FR-SEARCH-06 | 검색 반경 조절 가능 (5km / 10km / 20km) |

### 2.4 자동 전화 확인 (FR-CALL)
| 항목 | 내용 |
|------|------|
| FR-CALL-01 | 동시 3~5곳 응급실에 수용 여부 자동 전화 확인 |
| FR-CALL-02 | 기능 ON/OFF 토글 스위치 |
| FR-CALL-03 | 전화 확인 진행 상태 실시간 표시 |
| FR-CALL-04 | 확인 결과 즉시 지도/리스트에 반영 |

### 2.5 내비게이션 연동 (FR-NAVI)
| 항목 | 내용 |
|------|------|
| FR-NAVI-01 | 선택한 응급실 주소로 외부 내비 앱 연동 (카카오내비, T맵 등) |
| FR-NAVI-02 | 앱 내 간이 경로 미리보기 |

### 2.6 업무 로그 기록 (FR-LOG) — 구급대원 전용
| 항목 | 내용 |
|------|------|
| FR-LOG-01 | 상황 시작~종료 자동 타임스탬프 기록 |
| FR-LOG-02 | 환자 상태 정보 자동 포함 |
| FR-LOG-03 | 수용 병원 정보 (병원명, 주소, 전화번호) 기록 |
| FR-LOG-04 | 총 처리 시간 자동 산출 |
| FR-LOG-05 | 로그 파일 다운로드 (PDF / Excel) |
| FR-LOG-06 | 이전 출동 기록 열람 |

## 3. 비기능 요구사항

| 구분 | 요구사항 |
|------|----------|
| 성능 | 응급실 검색 결과 3초 이내 표시, LCP 2.5초 이내 |
| 가용성 | 24시간 365일 서비스 운영 (Vercel 글로벌 엣지 배포) |
| 보안 | 환자 정보 암호화 저장, **HTTPS 필수** (geolocation·mic 권한 요건), 서버 전용 키 분리 (`NEXT_PUBLIC_*` 외 모두 API Route 에서만 사용) |
| 접근성 | WCAG 2.1 AA 준수, 최소 48px 터치 영역, 고대비 색상, 음성 입력 |
| 호환성 | **Chrome / Edge 110+, Safari 15+, Samsung Internet 20+, Firefox 110+** — 모바일 우선 반응형, 데스크톱 확장 |
| PWA | Service Worker 로 앱 쉘·아이콘·최근 검색 결과 캐싱 → 홈화면 설치 가능. Android 는 추후 Bubblewrap 으로 TWA 패키징하여 Google Play 등재 가능 |
| 오프라인 | 앱 쉘·마지막 검색 결과·응급실 전화번호 로컬 캐싱(IndexedDB). 실시간 병상 정보는 네트워크 필수이며, 오프라인 시 "N분 전 데이터" 라벨 + 119 직접 전화 안내 |

## 4. 기술 아키텍처 (초안)

```
┌─────────────────────────────────────────────────────────┐
│          웹 클라이언트 (Next.js SSR/CSR, PWA)             │
│  ┌─────────┐  ┌──────────┐  ┌────────────────────────┐  │
│  │ 로그인   │  │ 환자입력  │  │  지도/리스트 표시         │  │
│  │ 모듈    │  │ 모듈     │  │  (Naver Maps JS v3)     │  │
│  └────┬────┘  └────┬─────┘  └────────┬───────────────┘  │
│       │            │                  │                  │
│  ┌────┴────────────┴──────────────────┴───────────────┐  │
│  │       클라이언트 서비스 레이어 (TanStack Query)       │  │
│  │       + Service Worker (오프라인 캐시)               │  │
│  └────────────────────┬──────────────────────────────┘  │
└───────────────────────┼─────────────────────────────────┘
                        │ HTTPS
┌───────────────────────┼─────────────────────────────────┐
│          Next.js API Routes / Server Actions              │
│  ┌────────────┐ ┌─────┴──────┐ ┌─────────────────────┐  │
│  │ 인증 프록시 │ │ 검색 프록시 │ │ 로그 API            │  │
│  │ (OAuth)    │ │ (서비스키보호)│ │ (dispatch/call)    │  │
│  └──────┬─────┘ └─────┬──────┘ └────────┬────────────┘  │
│         │             │                  │               │
│  ┌──────┴─────────────┴──────────────────┴────────────┐ │
│  │          Firebase (Auth + Firestore)                │ │
│  └────────────────────────────────────────────────────┘ │
└──────────────────────┬──────────────────────────────────┘
                       │
        ┌──────────────┼──────────────┐
        │              │              │
   ┌────┴────┐  ┌──────┴──────┐  ┌───┴────────────┐
   │ 공공     │  │ 네이버 지도  │  │ Web Speech API │
   │ 데이터   │  │ + Directions │  │ (브라우저 내장)│
   │ API      │  │  5 API      │  │                │
   └──────────┘  └─────────────┘  └────────────────┘
```

**핵심 설계 원칙**
- 공공데이터·네이버 Directions 등 **서비스 키가 필요한 외부 API는 반드시 Next.js API Route(서버)** 에서만 호출. 브라우저에 키를 노출하지 않는다.
- 지도·로그인 등 **클라이언트 키(JS SDK 전용)** 만 `NEXT_PUBLIC_*` 접두사로 브라우저에 노출한다.
- 세션은 Firebase Auth 의 ID Token을 HTTP-only 쿠키로 전달해 서버사이드 권한 검증에 사용한다.

## 6. 리스크 및 고려사항

| 리스크 | 대응 방안 |
|--------|-----------|
| 공공데이터 API 응답 지연/장애 | Next.js API Route 에서 TanStack Query staleTime + IndexedDB 캐싱 + "N분 전 데이터" 뱃지 |
| 자동 전화 기능 법적 이슈 | 서버사이드 Twilio 로 분리, 사전 동의 모달 + 법률 검토 |
| 음성 인식 정확도 (소음 환경) | Web Speech API `continuous: false` + 키워드 매칭 보정 + 수동 확인 단계 |
| 위치 권한 거부 시 | `navigator.geolocation` 에러 핸들링 → 수동 주소 입력 + IP 기반 대략 위치 폴백 |
| 개인정보 보호 (환자 정보) | 최소 수집 원칙 + Firestore 서버사이드 암호화 + 자동 삭제 정책 (PRD §2.6 참조) |
| HTTPS 미적용 환경 | Vercel 기본 HTTPS, 자체 서버 배포 시 Let's Encrypt 필수 (geolocation·mic·PWA 필요) |
| iOS Safari PWA 제약 | 설치 후 Web Push 부분 지원 — 배지 알림 대신 SMS/Email 폴백 검토 |
