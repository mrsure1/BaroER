# 바로응급실을 머신러닝 관점으로 이해하기

> 비전공자를 위한 친절한 가이드  
> 문서 버전: 1.0  
> 작성일: 2026-04-23  
> 주제: **바로응급실(BaroER)** 을 `y = f(x)` 관점에서 한 걸음씩 풀어보기

---

## 0. 왜 이런 문서를 읽어야 하나요?

바로응급실은 단순한 "병원 찾기 앱"처럼 보이지만, 내부적으로는 **"입력을 받아서 답을 내는"** 머신러닝의 사고방식과 완전히 같은 구조를 가지고 있습니다.

이 문서를 다 읽고 나면:

- "머신러닝에서 `y = f(x)`가 뭔가요?" 에 답할 수 있게 되고
- 바로응급실이 **실제로는 어떤 문제를 풀고 있는지** 기술적으로 설명할 수 있게 되고
- 앞으로 이 앱에 AI를 어떻게 붙일 수 있는지 감을 잡게 됩니다.

비전공자도 이해할 수 있도록 수식은 최소화하고, **일상 비유**를 많이 썼습니다.

---

## 1. `y = f(x)` 가 뭔가요? (3분 요약)

### 1.1 중학교 수학으로 시작

중학교에서 이런 걸 배웠던 기억, 있으시죠?

```
y = 2x + 3
```

- `x` 에 `5` 를 넣으면 → `y = 13`
- `x` 에 `10` 을 넣으면 → `y = 23`

여기서 **`f` 는 "규칙"**, **`x` 는 "입력"**, **`y` 는 "결과"** 입니다.

### 1.2 머신러닝은 이 규칙 `f` 를 **직접 찾아내는** 기술

수학 시간에는 `f` 를 사람이 만들어서 줬습니다. 머신러닝은 반대로:

> "입력 `x` 와 정답 `y` 가 잔뜩 있으니, **둘 사이의 규칙 `f` 를 컴퓨터가 스스로 발견하게 하자.**"

**일상 비유**: 엄마가 요리 레시피를 알려주지 않고, 재료(x)와 완성된 요리 사진(y) 수천 장만 보여주면서 "너가 알아서 규칙을 찾아봐"라고 하는 것. 이게 머신러닝입니다.

### 1.3 정리

| 기호 | 의미 | 예시 (키·몸무게로 비만도 예측) |
|------|------|------------------------------|
| `x` | 입력 (Input, Feature) | 키 170cm, 몸무게 75kg |
| `f` | 규칙 (Model) | "몸무게 ÷ (키²) 를 계산해서 판정" |
| `y` | 결과 (Output, Label, Target) | "정상 체중" |

## 2. 바로응급실의 `x` 는 무엇인가?

바로응급실 앱을 켰을 때, **앱에 들어가는 모든 정보**가 `x` 입니다.
정확히 세어 보면 **총 21개의 입력 변수(feature)** 가 있고, 이들을 **5개 그룹**으로 묶어 설명할 수 있어요.

> 요약: **x = (환자 정보 6개) + (상황·컨텍스트 4개) + (사용자 프로필 2개) + (병원별 실시간 속성 5개) + (경로·교통 4개) = 21개**

### 2.1 그룹 A. 환자 정보 — 사용자가 직접 입력 (**6개**)

| # | 변수 | 값 예시 | 입력 방식 |
|---|------|---------|-----------|
| x₁ | **증상 키워드** | 두통, 흉통, 외상, 호흡곤란, 복통, 의식저하 … | 체크리스트 선택 |
| x₂ | **상세 증상 텍스트** | "가슴이 10분째 답답해요" | 키보드 또는 음성(STT) |
| x₃ | **환자 성별** | 남 / 여 | 선택 |
| x₄ | **환자 연령대** | 영유아 / 소아 / 성인 / 노인 | 선택 |
| x₅ | **의식 상태** | 명료 / 기면 / 혼미 / 무반응 | 선택 |
| x₆ | **발병 시점** | "30분 전부터" | 텍스트 또는 선택 |

> 이 6개를 묶으면 **"환자가 지금 어떤 상태인가"** 를 설명하는 최소 정보가 됩니다.

### 2.2 그룹 B. 상황·컨텍스트 — 앱이 자동 수집 (**4개**)

| # | 변수 | 값 예시 | 수집 방법 |
|---|------|---------|-----------|
| x₇ | **현재 위치 (위도·경도)** | (37.5665, 126.9780) | `navigator.geolocation` |
| x₈ | **현재 시각 / 요일** | 2026-04-23 새벽 03:12 (평일) | 브라우저 시계 |
| x₉ | **검색 반경** | 5 km / 10 km / 20 km | 사용자 토글 |
| x₁₀ | **네트워크 상태** | 온라인 / 오프라인 / 느린망 | Service Worker · `navigator.onLine` |

### 2.3 그룹 C. 사용자 프로필 — 계정 기반 (**2개**)

| # | 변수 | 값 예시 | 수집 방법 |
|---|------|---------|-----------|
| x₁₁ | **사용자 유형** | 일반 시민 / 구급대원 | 회원가입 시 선택 |
| x₁₂ | **소속기관 코드** | "119-Seoul-Jongno" (구급대원만) | 기관 인증 |

### 2.4 그룹 D. 병원별 실시간 속성 — 공공데이터 API (**5개 × 병원 수**)

주변에 응급실이 N곳 있다면, **각 병원마다 아래 5개 속성이 붙어서** 입력됩니다.
(실무적으로는 `x₁₃` ~ `x₁₇` 이 병원 개수 N 만큼 반복되는 **행렬/리스트 형태**)

| # | 변수 | 값 예시 | 출처 |
|---|------|---------|------|
| x₁₃ | **응급실 병상 수 (일반/중환자/외상)** | 3 / 1 / 0 | 국립중앙의료원 응급의료정보 API |
| x₁₄ | **수용 가능 여부** | Y / 혼잡 / N | 공공데이터 실시간 피드 |
| x₁₅ | **대응 전문의 유무** | 응급의학과, 외과, 소아과, 산부인과 … | 공공데이터 |
| x₁₆ | **특수 장비 보유** | CT, MRI, 혈관조영실, ECMO, 수술실 가동 여부 | 공공데이터 |
| x₁₇ | **병원 기본정보** | 명칭, 주소, 전화번호, 응급실 좌표 | 공공데이터 (상수) |

### 2.5 그룹 E. 경로·교통 — 지도/내비 API (**4개 × 병원 수**)

| # | 변수 | 값 예시 | 출처 |
|---|------|---------|------|
| x₁₈ | **직선거리** | 3.4 km | 좌표 계산 |
| x₁₉ | **도로 기반 예상 소요시간 (ETA)** | 12분 | Naver Directions 5 |
| x₂₀ | **실시간 교통 상황** | 정체 / 원활 | Naver Directions 5 |
| x₂₁ | **자동 전화 확인 결과** (있을 경우) | 수용 ✅ / 불가 ❌ / 응답 없음 | Twilio 결과 피드백 |

> `x₂₁` 은 **전화를 돌린 후에 얻는 추가 입력**이라, 일반적인 최초 입력에는 없다가 **2차 이후에 갱신되는 동적 feature** 입니다.

### 2.6 한 줄(벡터)로 정리

```
x = ( x₁, x₂, x₃, x₄, x₅, x₆,          // 환자 정보 6개
      x₇, x₈, x₉, x₁₀,                   // 상황·컨텍스트 4개
      x₁₁, x₁₂,                           // 사용자 프로필 2개
      x₁₃, x₁₄, x₁₅, x₁₆, x₁₇,           // 각 병원별 실시간 속성 5개
      x₁₈, x₁₉, x₂₀, x₂₁ )               // 각 병원까지의 경로·교통 + 피드백 4개
```

즉 **"지금 이 순간의 응급 상황 스냅샷 = 21차원 feature 벡터"** 가 `x` 의 본질입니다.

### 2.7 비전공자용 짧은 비유

네비게이션 앱으로 비유하면:

- 그룹 A·B·C = **"내가 지금 어디서, 어떤 상태로, 어떤 자격으로"** 출발하려는지 (운전자 정보)
- 그룹 D = **"각 목적지(병원)의 현재 영업 상태"** (가게 오픈 여부·재고)
- 그룹 E = **"각 목적지까지 실제로 가는 길"** (거리·막힘 정도)

이 세 축이 합쳐져야 "어디로 가야 환자를 살릴 수 있는가"라는 `y` 가 계산됩니다.

### 2.8 (참고) 이 21개 중 지금 MVP가 실제로 쓰는 것과 나중에 채워질 것

| 지금 쓰는 것 | 데이터는 쌓지만 아직 모델이 안 씀 | 확장 예정 |
|---|---|---|
| x₁, x₂, x₃, x₄, x₅, x₇, x₉, x₁₁, x₁₃, x₁₄, x₁₇, x₁₈, x₁₉ | x₆, x₈, x₁₀, x₁₂, x₁₅, x₁₆, x₂₀, x₂₁ | 향후 Multi-task 모델로 `x₁~x₂₁` 전체를 통합 입력 |

## 3. 바로응급실의 `y` 는 무엇인가? — **이 문서의 핵심**

여기가 제일 재미있는 부분입니다. `y` 는 **"앱이 결국 우리에게 뭘 알려주고 싶은가?"** 입니다.

그런데 바로응급실의 `y` 는 **하나가 아니라 여러 층(Layer)으로 쌓여 있어요.** 하나씩 볼게요.

### 3.1 Layer 1. **"이 환자 얼마나 위급해?"** — 중증도 분류

```
y₁ = KTAS 등급 (1 ~ 5단계)
```

- **KTAS** 는 한국 응급환자 분류 기준이에요.
  - 1등급: 즉시 처치 (심정지, 중증 외상)
  - 5등급: 며칠 뒤에 봐도 되는 경증
- 앱은 환자 증상·의식·연령을 보고 **이 환자가 몇 등급인지 자동으로 판정**합니다.
- **머신러닝 용어**: *다중 클래스 분류 (Multi-class Classification)*
- **일상 비유**: 병원 접수대 간호사가 환자 얼굴을 보고 "중환자실 바로 가세요" vs "대기실에서 기다리세요"를 판단하는 것.

### 3.2 Layer 2. **"어느 병원이 제일 좋아?"** — 병원 순위 매기기 ⭐ **이게 핵심!**

```
y₂ = 추천할 응급실 목록 (1등, 2등, 3등, ...)
```

- 주변에 응급실이 10곳 있다면, **어디부터 가야 하는지 순서대로 줄 세우는 것**이 `y₂` 입니다.
- 각 병원마다 점수를 매겨서 정렬:

```
병원 점수 = (수용 가능성) × (환자 중증도에 맞는 장비) × (빨리 도착 가능한가)
         + (전문과 있음) - (현재 혼잡도)
```

- **머신러닝 용어**: *Learning-to-Rank (순위 학습)*
- **일상 비유**: 네이버에서 "파스타 맛집"을 검색하면 10개 중 가장 맛있을 것 같은 순서대로 보여주는 것과 똑같은 구조.

### 3.3 Layer 3. **"이 병원 지금 받아줄까?"** — 수용 가능성 예측

```
y₃ = 특정 병원이 이 환자를 받아줄 확률 (0% ~ 100%)
```

- 지도에 병원마다 🟢(수용가능) / 🟠(혼잡) / 🔴(불가) 색깔이 뜨는데, **이 색깔을 결정하는 숫자**가 바로 `y₃` 입니다.
- 앱이 "상위 3~5곳에 동시에 전화 확인"하기 *전에*, 모델이 "이 병원 받아줄 확률이 높네"를 미리 예측해서 **헛걸음을 줄입니다.**
- **머신러닝 용어**: *이진 분류 (Binary Classification)*
- **일상 비유**: 식당 예약 전에 "지금 이 시간에 자리 있을까?"를 감으로 먼저 판단하는 것.

### 3.4 Layer 4. **"얼마나 걸려?"** — 시간 예측

```
y₄ = 병원까지 예상 소요 시간 (분)
y₅ = 도착 후 치료 시작까지 대기 시간 (분)
```

- 리스트 화면에 "12분 / 3.4km" 같은 숫자가 바로 이 `y` 입니다.
- **머신러닝 용어**: *회귀 (Regression)* — 숫자를 예측하는 문제.

### 3.5 Layer ★. **진짜 최종 `y`** — 환자를 살릴 수 있는가

위의 `y₁` ~ `y₅` 는 사실 **중간 단계**에 불과합니다. 바로응급실이 **진짜 맞추고 싶은 답** 은:

```
Y★ = "골든 타임 안에 적절한 치료를 받게 할 수 있었는가?" (성공/실패)
     ≈ "환자가 살아서 집에 돌아갔는가?"
```

즉 이 앱의 존재 이유를 머신러닝 수식으로 쓰면:

```
f(환자 상태 x) = 환자의 생존 확률을 최대화하는 병원과 이송 경로
```

**일상 비유**: 네비게이션이 "가장 빠른 길"을 찾아주듯, 바로응급실은 **"환자를 가장 살릴 수 있는 길"** 을 찾아주는 내비게이션입니다.

## 4. 한 장짜리 정리표

### 4.1 입력 `x` — 총 21개 (5그룹)

| 그룹 | 개수 | 무엇? | 예시 변수 |
|------|------|-------|-----------|
| A. 환자 정보 | 6 | 증상·체크·기본정보 | x₁ 증상키워드, x₅ 의식상태 |
| B. 상황·컨텍스트 | 4 | 위치·시간·반경·네트워크 | x₇ GPS, x₈ 시각 |
| C. 사용자 프로필 | 2 | 유형·소속 | x₁₁ 일반/구급대원 |
| D. 병원 실시간 속성 | 5 × N | 병상·전문의·장비·수용여부 | x₁₄ 수용가능 |
| E. 경로·교통 | 4 × N | 거리·ETA·혼잡·통화결과 | x₁₉ 예상 소요시간 |

### 4.2 출력 `y` — 5개 층 + 1개 최종 목표

| 층 | 이름 | `y` 는 무엇? | 문제 유형 | 앱 어디에 보여지나? |
|----|------|--------------|-----------|---------------------|
| 1 | 중증도 판정 | KTAS 1~5 등급 | 다중분류 | 환자 입력 후 자동 표시 |
| 2 | **병원 추천 순위** | 추천 병원 리스트 | **Learning-to-Rank** | **리스트 화면 정렬 순서 (앱의 핵심)** |
| 3 | 수용 가능성 | 확률 0~100% | 이진분류 | 지도 🟢🟠🔴 마커 색깔 |
| 4 | 도착/대기 시간 | 분(minute) | 회귀 | 리스트의 "12분" 표시 |
| ★ | **환자 예후** | 생존/치료 성공 여부 | **의사결정 최적화** | **앱 전체의 존재 이유** |

## 5. 현재 MVP는 "학습"을 하고 있나요?

**답: 아직은 아닙니다.** 하지만 아주 중요한 준비를 하고 있어요.

### 5.1 지금은 **규칙 기반 (Rule-based)**

현재 앱은 사람이 정한 규칙으로 점수를 계산합니다:

```
점수 = 거리 × 0.4 + 수용가능 × 0.5 + 중증도매칭 × 0.1
```

이건 `f` 를 **개발자가 손으로 만든** 상태예요. 머신러닝이 아니라 그냥 **공식**입니다.

### 5.2 하지만 **데이터는 계속 쌓이고 있다**

앱이 돌아가는 동안, 특히 구급대원 출동 로그(`FR-LOG`) 기능이:

```
( 환자 상태 x , 실제로 이송한 병원 h , 처리시간, 결과 )
```

이런 쌍을 **자동으로 기록**하고 있어요. 이게 바로 **미래의 학습 데이터** 입니다.

### 5.3 충분히 쌓이면 "진짜 머신러닝 f"로 교체 가능

출동 기록이 수만 건 쌓이면:

1. "이런 환자에게는 이런 병원이 성공률이 높았다"는 **패턴을 컴퓨터가 스스로 발견**
2. 규칙 기반 점수를 **학습된 모델로 교체**
3. 병원 순위(`y₂`), 수용 예측(`y₃`), 시간 예측(`y₄`)이 훨씬 정확해짐

**이게 이 앱의 숨겨진 장기 전략입니다.** 지금은 규칙으로 시작하지만, 쓰면 쓸수록 똑똑해지도록 **데이터 파이프라인부터 먼저 깔아둔 것**.

## 6. 자주 나오는 질문 (FAQ)

### Q1. `y` 가 여러 개면 앱이 모델도 여러 개를 돌려야 하나요?
**A.** 네, 맞아요. 실제 서비스 머신러닝은 보통 **여러 작은 모델을 파이프라인처럼 연결**합니다.
예: 증상 → (모델A: KTAS 예측) → (모델B: 수용 확률 예측) → (모델C: 순위 정렬) → 화면 표시.

### Q2. 규칙 기반이면 AI 앱이 아닌가요?
**A.** 지금은 **AI 앱이라기보다 "AI-Ready 앱"** 입니다. AI를 *얹을 준비가 되어있는* 구조예요. 많은 성공한 서비스가 이 순서를 따릅니다 (규칙 → 데이터 수집 → ML 교체).

### Q3. 음성 입력도 머신러닝인가요?
**A.** 네, 그건 **Web Speech API** 라는 **이미 잘 만들어진 ML 모델(STT: Speech-to-Text)** 을 빌려 쓰는 경우입니다. 우리가 학습시키는 건 아니지만, 앱 안에서 동작하는 `f` 중 하나인 건 맞아요.
```
f_STT(음성 파형 x) = 글자 y
```

### Q4. "환자가 살았는가"를 `y` 로 학습하는 건 너무 잔인한 것 아닌가요?
**A.** 좋은 지적입니다. 실제로는 "생존"이라는 극단적 레이블 대신 **"골든타임 준수 여부", "첫 치료까지 걸린 시간", "재이송 없이 한번에 성공했는가"** 같은 **대리 지표(proxy label)** 로 학습합니다. 의료 ML 의 표준 방식이에요.

## 7. 한 줄 요약 (외워가세요)

> 바로응급실 = **21개의 입력 `x`** (환자 6 + 상황 4 + 프로필 2 + 병원 5N + 경로 4N) 을 받아서,
> **5층짜리 출력 `y`** (KTAS → 수용확률 → 순위 → 소요시간 → 환자 예후) 를 내는 **`f`** 를 만드는 프로젝트.

> 지금의 `f` 는 사람이 짠 규칙, 미래의 `f` 는 데이터로 학습된 머신러닝 모델.
