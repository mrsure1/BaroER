# 3C 분석 — 바로응급실(BaroER) 배포 전략

> 작성일: 2026.04.24 · 작성자: 이진영(미스터슈어, 팀 AML)
> 형식: 3C(Customer / Company / Competitor) + USP 한 줄 정의
> 목적: 파일럿 배포(Beta Launch) 이전 **"누구에게 · 무엇을 · 왜 우리에게서"** 를 1페이지로 확정한다.

---

## 0. 요약 — "한 줄 앱 정의" (USP)

> **"부모님이 갑자기 쓰러졌을 때, 앱 설치 없이 링크 한 번으로 가장 빨리 수용 가능한 응급실을 찾아주는 음성 기반 응급의료 PWA."**

- **핵심 타겟**: 60세 이상 부모를 둔 30·40대 맞벌이 자녀 (1차)  ·  현장 구급대원 (2차)
- **핵심 문제**: 응급 상황에서 "어느 응급실로 가야 하는가?" 의 결정 지연 (= 골든타임 소실)
- **핵심 기능 + 차별 포인트**: 음성 → KTAS 자동 분류 → 실시간 수용 가능 응급실 → 자동 전화 → 내비까지 **하나의 화면 흐름**으로 완결되는, **설치 없는 PWA**.

## 1. 🟦 Customer — 가장 중요한 타겟 고객 1명

### 페르소나 "박지영, 38세 · 서울 마포 거주 · IT 마케터 · 자녀 1명 · 양가 부모 모두 60대"

#### 1-1. 어떤 사람인가? (나이 · 직업 · 상황)

| 항목 | 내용 |
|---|---|
| **나이/성별** | 38세 여성 (또는 30대 중반 남성) |
| **직업/소득** | 정규직 사무·기술직, 가구 소득 연 7,000만~1.2억 |
| **거주** | 수도권·광역시 아파트, 부모는 지방 중소도시 단독 거주 |
| **디지털 숙련도** | 상: 카카오T·배민·토스 능숙, PWA 개념 몰라도 "홈 화면 추가" 거부감 없음 |
| **건강 상태** | 본인 건강 양호. 양가 부모 중 **최소 1인이 만성질환(고혈압·당뇨·부정맥)** 보유 |
| **가족 구조** | 맞벌이 + 미취학·초등 자녀 1~2명 → 시간 희소성 극대 |
| **스마트폰 사용 시간** | 평균 4~5시간/일, 주로 iOS 또는 Android 최신 2세대 |
| **정보 습득 채널** | 유튜브 > 네이버 블로그·카페 > 인스타그램 릴스 > 지인 SNS |

#### 1-2. 가장 큰 Pain Point 한 문장은?

> **"부모님이 쓰러졌다는 전화를 받은 그 순간, '어느 응급실로 모셔야 하는지' 를 검색·전화·판단까지 혼자서 3분 안에 끝내야 한다."**

세부 고통:
- 119에 전화하면 병원을 사용자가 정해야 하는 경우가 많음 (관할·수용 가능성 판단 어려움)
- "E-Gen 포털" 은 PC 중심 UI · 모바일에서 수용 가능성 파악 어렵고 데이터도 실시간 인식이 어려움
- "응급똑똑" 은 "어떤 증상인지" 알려주는 앱이지, "어디로 갈지" 알려주지 않음
- 밤/주말에는 야간·주말 진료 병원 판별이 특히 혼란스러움
- 부모·보호자는 **패닉 상태** → 손이 떨려 타이핑 어려움 (음성 입력이 결정적)

#### 1-3. 언제 이 앱이 필요한가? (JTBD · Job to Be Done)

> **"[When] 부모님·가족이 갑작스런 흉통·의식 저하·낙상을 보일 때**
> **[I want to] 가장 가까운 '수용 가능한' 응급실을 30초 안에 찾아 전화까지 연결하고 싶다**
> **[So I can] 내가 직접 도착하기 전에 이송·수용 의사결정을 완료해 골든타임을 지킬 수 있다."**

**대표 트리거 상황(4가지)**
1. **주말 오후, 아버지의 급성 흉통** — 대학병원 응급실 전화해도 계속 "만실"
2. **야간, 어머니 낙상으로 의식 저하** — 동네 병원은 야간 미운영, 대학병원은 수용 불가
3. **출근 중 가족의 긴급 연락** — 본인은 지하철, 원격으로 결정 지원 필요
4. **여행지에서 동행자 급환** — 낯선 지역, 관할·수용 응급실 정보 전무

## 2. 🟩 Company — 우리 앱의 핵심 강점 · 약점

#### 2-1. 경쟁 대비 가장 강한 기능 1가지는?

> **"음성 → KTAS 자동 → 실시간 수용 가능 응급실 → 자동 전화 → 내비" 를 '하나의 화면 흐름'으로 끊김 없이 연결하는 유일한 서비스.**

근거
- **E-Gen**: 정보 열람에 멈춤. 전화·내비·분류 통합 안 됨.
- **응급똑똑**: 증상 상담이 엔드포인트. "어디로 갈지" 의사결정은 사용자 몫.
- **우리**: **단 한 화면·단 한 흐름** 으로 "증상 인식 → 병원 선택 → 전화 → 출발" 까지 **의사결정을 닫는다** = 경쟁의 본질적 재정의.

강점 Top 5 (중요도 순)
1. **흐름의 완결성** — 판단 지연을 없애는 단일 화면 경험
2. **PWA 즉시 실행** — 설치 장벽 0, 링크 공유 = 최고의 바이럴
3. **KTAS 표준 기반 중증도 자동 분류** — 초기 신호를 시스템이 제시해 패닉 상황의 의사결정 부담 감소
4. **구급대원 전용 모드** (국내 유일의 업무 로그 자동화 PDF/Excel)
5. **ML-Ready 로그 파이프라인** — 누적 데이터로 "환자 상태 × 수용 가능성 예측 모델" 고도화 경로

#### 2-2. 파일럿 기간 내 반드시 개선해야 할 약점?

| 약점 | 영향 | 파일럿 내 대응 |
|---|---|---|
| **W1. 브랜드 인지도 0** | 검색·발견성 부족 | 네이버 블로그·카페·유튜브 Shorts 바이럴 + "부모님 핸드폰에 저장하세요" 캠페인 |
| **W2. 자동 전화(Twilio)의 법적·윤리 리스크** | 무단 전화 신고·민원 가능 | 사전 동의 모달 명시, 콜 시 "바로응급실 자동 연결 서비스입니다" 고지, 로그·녹취 정책 공개 |
| **W3. STT 정확도 (소음·방언)** | 오입력 시 KTAS 잘못 추정 | 이모지·태그 백업 입력, "방금 말한 것 맞나요?" confirm 스텝, 로그 수집 후 규칙 엔진 튜닝 |
| **W4. 공공데이터 API 지연·장애** | 핵심 UX 신뢰도 훼손 | Service Worker 캐시 + "N분 전 데이터" 라벨 + IndexedDB 폴백으로 "최소 정보" 보장 |
| **W5. 1인 팀의 운영 커버리지** | 장애 대응·지원 한계 | 장애 자동 알림(Sentry·Slack) + "문의→지인 CS 에이전트 위임" + 파일럿 규모 제한 (동시 활성 500명 cap) |
| **W6. iOS PWA 푸시 한계** | 재방문 약화 | iOS 16.4+ 웹푸시 대응 + 이메일·카카오톡 공식 채널 병행 |

#### 2-3. 핵심 기능 1개에 집중한다면?

> **"실시간 수용 가능 응급실 지도·리스트 + 자동 전화 확인"** — 즉, **"어디로 가야 하는가?" 에 대한 답을 30초 안에 '전화 연결된 상태'로 제공하는 것**.

이유
- 페르소나의 Pain Point(1-2) 와 정확히 매칭되는 단일 기능
- 경쟁이 가장 약한 영역 (E-Gen은 열람, 응급똑똑은 상담)
- 이 한 기능만으로 **"링크 공유 → 사용 → 가족에게 공유"** 의 바이럴 루프가 자동 발생
- 나머지 기능(온보딩·KTAS·구급대원 모드)은 이 중심 기능의 **부스터**로 포지셔닝

## 3. 🟪 Competitor — 사용자가 지금 쓰는 대안

#### 3-1. 직접 경쟁 앱 Top 2

| # | 서비스 | 제공 주체 | 포지션 | 사용자 수(추정) |
|---|---|---|---|---|
| **① E-Gen** | 보건복지부·국립중앙의료원 | 공공 포털 — 응급실 정보 열람 중심 | 높음 (공공 브랜드) |
| **② 응급똑똑** | 국립중앙의료원 | 민간 보급 앱 — 증상 안내·상담 중심 | 중 |

#### 3-2. 사용자가 지금 쓰는 **간접 대안** (진짜 경쟁자)

| # | 대안 | 사용자 행동 |
|---|---|---|
| **① 119 전화 + 상담원 안내** | "일단 119 걸고 상담원한테 맡긴다" — 가장 흔한 디폴트 |
| **② 네이버·구글 '응급실 + 지역명' 검색** | 실시간 수용 여부는 모름, 전화번호만 확보 |
| **③ 카카오맵/네이버지도 '응급실' 필터** | 거리만 보임, 수용 가능성·병상·전문의 정보 없음 |
| **④ 가족·지인 전화 상담 ("의사 친구 없어?")** | 심리적 안도 높음, 의학적·지역적 정확성 낮음 |
| **⑤ 지역 맘카페·오픈채팅방 문의** | 속도 느림, 밤/새벽 응답 부재 |

**전략적 시사점**: 우리의 진짜 적은 E-Gen/응급똑똑이 아니라 **"119 전화 + 검색 + 지도 앱"의 파편화된 행동 조합**. 이걸 **한 화면**으로 흡수하는 것이 포지셔닝의 본질.

#### 3-3. 그 대안의 가장 큰 불만은?

> **"정보가 여러 곳에 흩어져 있고, 정작 '지금 이 병원이 받아 주는가' 를 알려주는 곳이 없다."**

구체적 불만 5선 (실제 맘카페·블로그·E-Gen 앱스토어 리뷰 조사 기반 요약)
1. **"실시간 수용 여부가 없음"** — E-Gen도 데이터 지연, 전화 돌려봐야 확인
2. **"도착하고 보니 진료 불가"** — 가장 치명적인 실패 경험
3. **"모바일 UX가 공공 포털 느낌 · 어르신은 못 씀"** — 큰 글씨·음성 입력 부재
4. **"앱 깔아야 함"** — 응급 상황에서 설치·회원가입·OTP 인증 = 금기
5. **"구급대원이 쓰는 도구와 시민용이 따로"** — 이송 중 병원 확정 정보 공유 단절

**우리의 즉답**: 5개 중 4개(①②④⑤)는 바로응급실의 **구조적 기본값**으로 해결, ③은 온보딩·접근성 개선으로 파일럿 내 해결.

## 4. 🟨 USP — 한 줄 앱 정의 완성

#### 4-1. 핵심 타겟

> **"60세 이상 부모를 둔 30·40대 맞벌이 자녀"** (1차 타겟, Beachhead)
> + **"현장 출동 구급대원"** (2차 타겟, 수직 확장)

#### 4-2. 핵심 문제

> **"부모님·가족의 응급 상황에서 '어느 응급실이 지금 받아 주는가' 를 30초 안에 결정할 방법이 없다."**

#### 4-3. 핵심 기능 + 차별 포인트 (한 문장)

> **"음성으로 증상을 말하면 KTAS가 자동 분류되고, 실시간 수용 가능 응급실을 색상으로 보여주며, 탭 한 번에 동시 3~5곳에 자동 전화를 걸어주는, 설치가 필요 없는 PWA."**

#### 4-4. 1줄 포지셔닝 스테이트먼트

> **For** 부모님 응급 상황을 혼자 판단해야 하는 30·40대 자녀,
> **BaroER is** 설치 없이 링크 한 번으로 실행되는 응급의료 PWA,
> **that** 음성·KTAS·실시간 공공데이터·자동 전화를 하나의 화면 흐름으로 연결해 30초 내 "어디로 갈지" 를 결정시켜 준다.
> **Unlike** E-Gen/응급똑똑/네이버지도·119 전화의 파편화된 경험,
> **we** "판단 자체를 끝내 주는" 유일한 서비스다.

## 5. 배포 전략 시사점 (3C → Go-To-Market)

### 5-1. 채널 전략 (Where to Launch)

| 단계 | 채널 | 메시지 | KPI |
|---|---|---|---|
| **Phase 0. Seed (출시 전 2주)** | 개인 SNS · 사내 · 친지 | "부모님 핸드폰에 꼭 저장하세요" | 베타 대기자 300명 |
| **Phase 1. Viral Seed (출시 ~ 4주)** | 유튜브 Shorts · 인스타 릴스 · 맘카페 "부모님 돌봄" 게시판 | 제작한 62초 공식 영상 + 실제 사용 시연 Shorts | 누적 조회 10만 / 유입 3,000 UV |
| **Phase 2. Public Signal (4 ~ 8주)** | IT·의료 전문 미디어(플래텀·테크42·청년의사) | "1인 개발자가 만든 공공 응급의료 PWA" 보도자료 | 기사 5건, 공식 트래픽 배수 |
| **Phase 3. 공공 연계 (8주 ~ )** | 소방청·지자체·보건소 협력 제안 | 구급대원 모드 실증 + 오픈소스 공개 | MOU 1~2건, 파일럿 지자체 확보 |

### 5-2. 메시지 프레임 (What to Say)

**1줄 메인 카피 (유튜브 제목, 썸네일, 첫 화면)**
> "응급실 뺑뻉이, 이제 그만. 설치 없이 링크 한 번으로."

**보조 카피 (소셜 공유, 검색)**
- "부모님이 쓰러졌을 때, 30초 안에 병원을 찾는 방법."
- "119 전화와 네이버 지도 사이에 있던 그 공백을 채웠습니다."
- "설치 X · 즉시 실행 ✓ · 음성 한 번으로 응급실."

### 5-3. 회피해야 할 메시지 (Don't)

- ❌ **"AI 응급의료 혁신"** — 추상적, 페르소나 정서 미매치
- ❌ **"의사 상담 가능"** — 법적 리스크, 기능 정렬 안 됨
- ❌ **"실시간 병상 수 100% 정확"** — 공공데이터 지연 리스크로 부정확 주장 불가
- ❌ **"119 대체"** — 규제·신뢰 리스크, 119 "보조" 포지셔닝 유지

### 5-4. 파일럿 지표 (90일 목표)

| 지표 | 목표 |
|---|---|
| 총 방문 (UV) | 30,000 |
| 첫 검색까지 이르는 전환율 | 40% |
| 평균 의사결정 시간 (증상 입력 → 전화 시도) | < 45초 |
