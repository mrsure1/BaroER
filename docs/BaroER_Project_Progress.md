# 바로응급실 (BaroER) - 프로젝트 수행 경과 및 자체 평가

요청하신 **[4. 프로젝트 수행 경과]** (3~5장 분량)와 **[5. 자체 평가 의견]**에 대한 슬라이드 기획안입니다.

---

## 4. 프로젝트 수행 경과

### [Slide 1] 4-1. 기획 단계 및 기술 아키텍처 수립
**"사용자 접근성과 개발 효율성을 고려한 플랫폼 전환 및 설계"**
*   **초기 요구사항 정의 및 PRD 작성:**
    *   응급실 만상 문제를 해결하기 위한 실시간 데이터 탐색 요구사항 정의
    *   일반 사용자 및 구급대원(119) 각각의 User Flow 설계
*   **플랫폼 전략 전환 (React Native → PWA):**
    *   초기에는 네이티브 앱(React Native)으로 기획하였으나, 긴급 상황 시 앱 설치 없이 즉시 브라우저로 접근할 수 있는 **Next.js 기반 PWA(Progressive Web App)** 로 기술 스택 전면 개편.
*   **화면 및 DB/API 설계:**
    *   직관적인 모바일 우선(Mobile-First) UI/UX 화면 설계 완료
    *   확장성을 고려한 Supabase (PostgreSQL) 기반 DB 스키마 및 REST API 설계

### [Slide 2] 4-2. 프론트엔드 기반 구축 및 데이터 연동
**"안정적인 인프라 구축 및 외부 공공데이터 통합"**
*   **Next.js 스캐폴딩 및 디자인 시스템 적용:**
    *   Tailwind CSS를 활용하여 일관된 디자인 토큰 및 재사용 가능한 UI 컴포넌트 개발
*   **Supabase 인증 및 백엔드 연동:**
    *   이메일 및 소셜(Google, Kakao) 로그인 구현
    *   JWT 토큰 및 HTTP-only 쿠키를 활용한 안전한 세션 관리 시스템 구축
*   **공공데이터 포털 API 프록시 서버 구축:**
    *   실시간 응급실 수용 정보 조회를 위한 외부 API 연동
    *   CORS 문제 해결 및 API Key 보안을 위해 브라우저 직접 호출을 차단하고, Next.js API Routes를 경유하는 안전한 프록시 서버 설계

### [Slide 3] 4-3. 핵심 기능(지도, 음성) 및 UI 구현
**"긴급 상황에 최적화된 사용자 인터페이스 완성"**
*   **위치 기반 응급실 탐색 및 시각화:**
    *   Naver Maps API를 연동하여 내 위치 주변(5/10/20km) 응급실 마커 표시
    *   응급실의 실시간 수용 가능 여부를 3가지 색상(🟢/🟠/🔴)으로 직관적 매핑
*   **음성 인식(STT) 및 환자 상태 입력:**
    *   Web Speech API를 활용하여 긴급 상황에서 텍스트 타이핑 없이 음성으로 증상을 입력할 수 있는 기능 구현
    *   입력된 증상을 바탕으로 한 중증도 자동 분류 로직 적용
*   **내비게이션 딥링크 연동:**
    *   목적지 병원을 선택하면 카카오내비, T맵 등으로 즉시 길안내가 시작되도록 URL 스킴(Scheme) 연동

### [Slide 4] 4-4. 구급대원 특화 기능 및 업무 자동화
**"119 구급대원의 행정 업무 부담을 줄이는 맞춤형 기능"**
*   **자동 전화 확인 시스템 개발:**
    *   응급실 뺑뺑이를 막기 위해, 타겟 병원 3~5곳에 수용 가능 여부를 자동 발신하여 확인하는 백엔드(Twilio 연동 등) 로직 설계 및 테스트
*   **구급대원 전용 업무 로그 자동화:**
    *   출동 시작부터 환자 이송 완료까지의 타임스탬프, 환자 상태, 병원 정보 자동 기록
*   **리포트 생성 및 내보내기 기능:**
    *   완료된 출동 로그를 기반으로 현장 보고서를 자동 구성하고, PDF 및 Excel 포맷으로 다운로드하는 기능 구현

### [Slide 5] 4-5. 최적화, PWA 패키징 및 최종 배포
**"앱과 동일한 사용자 경험 제공 및 서비스 런칭"**
*   **PWA 적용 및 오프라인 캐싱 (Service Worker):**
    *   모바일 기기 홈 화면에 아이콘 추가(설치) 기능 활성화
    *   네트워크 불안정 시에도 최근 검색 기록 및 기본 UI가 동작하도록 IndexedDB와 Service Worker를 활용한 캐싱 처리
*   **글로벌 엣지 배포 (Vercel):**
    *   Vercel 환경에 최적화하여 무중단 CI/CD 파이프라인 구축 및 배포 완료
*   **성능 및 접근성 최적화:**
    *   LCP(가장 큰 콘텐츠 렌더링) 2.5초 이내 달성 및 터치 영역 최적화(WCAG 2.1 준수)
    *   (추후 진행) Bubblewrap을 활용한 TWA(Trusted Web Activity) 패키징 후 Google Play Store 등재 준비

## 5. 자체 평가 의견

### [Slide 6] 5. 자체 평가 의견 및 향후 발전 방향
**"실효성 높은 플랫폼 구축 완료, 데이터 및 AI 기반 서비스로의 도약 준비"**

*   **잘된 점 (Strengths):**
    *   초기 네이티브 앱에서 PWA로 과감하게 전환하여, 응급 상황에서 '앱 설치'라는 진입 장벽을 없애고 즉각적인 접근성을 확보한 점이 가장 큰 성과입니다.
    *   공공데이터 API와 자체 구축한 프록시 서버를 통해 실시간 데이터를 안정적으로 시각화(지도/리스트)하여 직관성을 높였습니다.
    *   음성 입력 및 구급대원 업무 자동화(로그/PDF 생성) 기능을 통해 실제 사용자의 Pain Point를 정확히 해결했습니다.
*   **아쉬운 점 (Weaknesses/Challenges):**
    *   공공데이터 API 자체의 응답 지연이나 데이터 누락이 발생할 경우, 앱의 신뢰도에 영향을 줄 수 있어 자체 캐싱 및 예외 처리(Fallback) 로직 보완이 추가로 필요했습니다.
*   **향후 발전 방향 (Future Plans):**
    *   단순 현재 수용 여부를 넘어, 누적된 환자 데이터를 활용해 **'시간대별/증상별 응급실 수용 확률 예측 AI 모델'** 도입을 검토 중입니다.
    *   서비스 안정화 후 Google Play Store 배포를 통해 더 많은 사용자에게 서비스를 제공할 계획입니다.
