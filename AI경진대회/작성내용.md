  한 화면에서 수행하는 현장 행동형 응급의료 에이전트.

### 2. 기술 명세

- **2.1 기술목적**
  1. 응급 상황에서 가장 가까운 "수용 가능" 응급실을 **3초 이내에 찾아주는 실시간 공공데이터 기반 의사결정 지원 도구** 제공.
  2. 음성·증상 키워드 기반으로 **KTAS 중증도(Level 1~5)를 자동 분류**하여 환자 수용 판단의 초기 신호를 제공.
  3. 구급대원의 업무 로그(출동~이송~완료)를 자동 기록·집계해 **행정 부담 경감 및 통계적 운영 개선** 지원.
  4. 장기적으로는 누적 로그를 학습 데이터로 활용해 **환자 상태(X) → 수용 가능 응급실 추천(Y) 예측 모델**로 발전.

- **2.2 기술구조** (PWA 클라이언트 + 서버사이드 프록시 + 공공·지도·음성 API 통합)
  ```
  [브라우저 PWA 클라이언트 (Next.js 15 + React 19)]
   ├─ 인증/온보딩 모듈 (Firebase Auth)
   ├─ 증상 입력 모듈 (텍스트 + Web Speech API STT)
   ├─ KTAS 중증도 분류기 (초기: 규칙 기반, 향후: 경량 ML)
   ├─ 지도·리스트 뷰 (Naver Maps JS v3 + TanStack Query)
   └─ Service Worker (PWA 오프라인 캐시, IndexedDB)
           │  HTTPS
  [Next.js API Route / Server Actions]
   ├─ 공공데이터 프록시 (응급의료정보센터 API, 서비스키 보호)
   ├─ 내비 URL 스킴 생성 (KakaoNavi / Tmap / Naver Maps)
   ├─ 자동 전화 확인 (Twilio 서버리스)
   └─ 로그 API (Firestore 저장)
           │
  [Firebase(Auth/Firestore) · 공공데이터 API · Naver Maps · Twilio]
  ```

- **2.3 주요기능**
  1. **음성 + 키워드 기반 증상 입력** (Web Speech API, 권한 없이 수동 입력으로 폴백)
  2. **KTAS 자동 중증도 분류** (호흡곤란·흉통·의식저하 등 레드플래그 매칭)
  3. **실시간 응급실 탐색** (공공데이터 API 기반 수용 가능 여부, 병상 수, 전문의 유무)
  4. **지도 / 리스트 듀얼 뷰** (🟢수용가능 · 🟠혼잡 · 🔴불가 색상 마커)
  5. **동시 다발 자동 전화 확인** (동시 3~5곳, 진행상황 실시간 표시)
  6. **외부 내비 앱 연동** (KakaoNavi · Tmap · Naver Maps URL 스킴)
  7. **구급대원 업무 로그 자동화** (타임스탬프·환자상태·수용병원·총처리시간 PDF/Excel 내보내기)
  8. **PWA 홈화면 설치 + 오프라인 최근 검색 캐시**

- **2.4 결과물 형상**
  - **Progressive Web App (PWA)** — 모바일/데스크톱 브라우저에서 즉시 실행, 홈화면 설치 시 네이티브 앱 수준의 UX 제공.
  - 향후 Google Bubblewrap로 **TWA 패키징해 Google Play 등재 가능**.

- **2.5 배포방식**
  - **Vercel 글로벌 엣지 네트워크**를 통한 SSR/CSR 하이브리드 배포 (HTTPS 기본 제공).
  - 공공데이터·Naver Directions 등 **서비스 키가 필요한 API는 Next.js API Route(서버)에서만 호출**해 키 노출 방지.
  - 소스코드는 GitHub 공개 저장소로 관리하며, 대회 기간 중 심사위원에게 초대 권한 부여.

- **2.6 혁신적 요소**
  1. **설치 없이 즉시 실행**되는 PWA로, 골든타임을 다투는 응급 상황에서 "앱 다운로드" 장벽 제거.
  2. **음성 → KTAS → 자동 전화 → 내비**까지 4단계를 **하나의 화면 흐름**으로 통합 (국내 유사 앱 대비 최초).
  3. **구급대원 전용 업무 로그**(타임스탬프 자동 기록 + PDF/Excel 보고서) — 119 현장 워크플로우를 직접 반영.
  4. **오프라인 폴백**(Service Worker + IndexedDB + "N분 전 데이터" 라벨)으로 네트워크 단절 상황에서도 최소 정보 제공.
  5. **ML-Ready 데이터 파이프라인** — 모든 입력·선택·결과를 익명 로그로 축적해 향후 학습 자산으로 전환.

- **2.7 도전적 요소**
  1. **다중 공공 API + 자동 전화 + STT + 지도 + 내비**를 1인 개발로 End-to-End 통합하는 기술 복합성.
  2. **Twilio 기반 자동 전화 확인**의 법적·운영적 리스크 관리 (사전 동의 모달, 서버사이드 분리).
  3. **소음 환경의 STT 정확도 + KTAS 레드플래그 자동 분류**의 의료 도메인 정확도 확보.
  4. **실시간 공공데이터 장애·지연**을 사용자가 체감하지 못하도록 하는 캐시·폴백 전략.

### 3. 구현 방법 및 계획

- **3.1 구현범위 (3개 이상 세부업무)**
  - **세부업무 #1 — 인증 · 온보딩 · 증상 입력 + KTAS 자동 분류**
    Firebase Auth 기반 이메일/구글 로그인, 사용자 유형(일반/구급대원) 분기, 최초 1회 온보딩 가이드,
    Web Speech API STT, 증상 키워드·레드플래그 규칙 엔진을 통한 KTAS Level 자동 추정.

### 4. 파급효과

- **4.1 기술적 파급효과**
  1. **공공 응급데이터 × 음성 STT × 자동 전화 × KTAS 분류**를 통합한 국내 최초의 현장 행동형 PWA 레퍼런스 제시.
  2. **서비스 키 보호(API Route) + 오프라인 폴백 + ML-Ready 로그 파이프라인**을 포함한 **공공데이터 기반 PWA 설계 패턴**을 오픈소스로 공개해, 타 공공·안전 분야(소방·재난·보건)로 재사용 가능.
  3. 축적된 익명 로그를 바탕으로 **"환자 상태 x 병원 수용 가능성 예측"** 의료 AI 모델 논문·특허 출원 가능.

- **4.2 사회·산업적 파급효과**
  1. **응급실 뺑뻉이 최소화 및 골든타임 단축** — 심정지·뇌졸중·외상 등 중증 환자의 생존율 향상에 직접 기여.
  2. **119 구급대원의 행정 부담 경감** — 업무 로그 자동화로 출동당 보고서 작성 시간을 큰 폭으로 단축.
  3. **국민 누구나 링크 한 번으로 사용**할 수 있는 공공서비스 형태로 제공 가능 → 소방청·지자체·보건소 협력 모델 기대.
  4. **응급의료 시스템의 디지털 전환**을 촉진하며, 통계 대시보드·재난 대응 API로 확장 가능.
